# Spark Retail Case Study — Design

**Project:** Big Data Analytics Mini Project 2 — Spark Case Study
**Dataset:** Online Retail II (UCI ML Repository) — `online_retail_II.csv`, 1,067,372 rows, 8 columns
**Date:** 2026-04-25

---

## 1. Problem Statement

**Research question:** *Which customer segments drive revenue, and how do their purchasing patterns evolve over time and across countries?*

This combines two classical online-retail analytics angles:

1. **RFM customer segmentation** — classify each customer by Recency, Frequency, and Monetary value to identify Champions, Loyal, At-Risk, and Lost customers.
2. **Sales trend analysis** — uncover monthly seasonality, country-level patterns, and month-over-month growth.

The combined question requires per-customer aggregation, time-windowed aggregation, and joins between the two — exactly the workload pattern that justifies Spark's distributed Structured APIs over single-machine analytics.

## 2. Dataset Description

| Property | Value |
|---|---|
| Source | UCI ML Repository — Online Retail II |
| File | `online_retail_II.csv` |
| Rows | 1,067,372 |
| Columns | 8 |
| Size | ~95 MB on disk (CSV) |
| Time range | 2009-12-01 to 2011-12-09 |

**Schema:**

| Column | Type | Notes |
|---|---|---|
| `Invoice` | string | Invoice number; values starting with `'C'` are cancellations |
| `StockCode` | string | Product code |
| `Description` | string | Product name |
| `Quantity` | integer | Negative for returns/cancellations |
| `InvoiceDate` | timestamp | Format `yyyy-MM-dd HH:mm:ss` |
| `Price` | double | Unit price in GBP |
| `Customer ID` | double (nullable) | Anonymized customer identifier |
| `Country` | string | ~40 distinct values |

**Justification for Spark:** the dataset has > 1M rows, mixed types (numeric, string, timestamp, categorical), and supports joins that benefit from broadcast vs sort-merge optimization. While 95 MB fits in memory on a single machine, the analytical patterns (group-bys with high-cardinality keys, window functions over millions of rows, multi-stage aggregations) demonstrate Spark's distributed execution model. The dataset is also a canonical example used in academic and industry tutorials for customer-segmentation workloads.

## 4. Queries

### Mapping to the rubric's required query categories

| # | Query | Rubric requirement |
|---|---|---|
| Q1 | Data cleaning filter | (1) Filtering with complex conditions |
| Q2 | Top-line aggregates | (2) SUM, AVG, COUNT, MAX/MIN |
| Q3 | Revenue by Country × Year × Month | (3) Group by multiple attributes |
| Q4 | Top-N products by revenue | (4) Sorting and ranking |
| Q5 | Monthly cumulative revenue + 3-month moving avg | (5) Window — cumulative & moving avg |
| Q6 | Customer rank within country | (5) Window — ranking |
| Q7 | Above-average customers | (6) Nested/subquery |
| Q8 | Country → Region join (broadcast hint) | (7) Broadcast join optimization |
| Q9 | Customer × month self-join (sort-merge) | (7) Sort-merge join optimization |
| Q10 | RFM segmentation | (5) Window NTILE; project headline insight |

### Detailed specifications

**Q1 — Cleaning filter.** Keep rows where `Quantity > 0` AND `Price > 0` AND `Customer ID IS NOT NULL` AND `Invoice NOT LIKE 'C%'` AND `InvoiceDate BETWEEN '2009-12-01' AND '2011-12-09'`. Output: cleaned row count and the filtered DataFrame (used as input by downstream queries).

**Q2 — Top-line aggregates.** Single-row result with: total revenue (`SUM(Quantity*Price)`), distinct invoice count, distinct customer count, average order value, max single-line revenue, min/max `InvoiceDate`.

**Q3 — Revenue by Country × Year × Month.** Group by `(Country, year(InvoiceDate), month(InvoiceDate))`, output `revenue` and `order_count` per cell. Sort by revenue descending. Used later for both the CSV-vs-Parquet and partition-pruning demos.

**Q4 — Top-10 products by revenue.** Group by `(StockCode, Description)`, sum revenue, sort descending, limit 10.

**Q5 — Monthly trend.** Aggregate revenue per `(year, month)`, then a window over the time axis to compute cumulative revenue and a trailing 3-month moving average (current month plus the two previous months). Trailing rather than centered so the latest month's value is computable.

**Q6 — Customer rank within country.** Per-customer total spend, then `ROW_NUMBER() OVER (PARTITION BY Country ORDER BY total_spend DESC)`. Output: top 5 customers per country.

**Q7 — Above-average customers (subquery).** SQL form: `SELECT customer_id, total_spend FROM customer_totals WHERE total_spend > (SELECT AVG(total_spend) FROM customer_totals)`. The DataFrame form uses a scalar aggregate. The RDD form materializes the average then filters.

**Q8 — Country → Region join (broadcast).** A small in-memory map (~40 rows) of `Country → Region` (Europe / Americas / Asia / Other) joined with the cleaned sales table to produce regional revenue. The DataFrame and SQL forms use `broadcast(region_df)` / `/*+ BROADCAST(region_df) */` so the physical plan is a `BroadcastHashJoin`.

**Q9 — Customer × month self-join (sort-merge).** First aggregate per `(customer_id, month_index)` where `month_index = year * 12 + month`. Then self-join on `customer_id` and `month_index = month_index_other - 1` to compute month-over-month spend delta per customer. Both sides are large, neither broadcastable; physical plan is `SortMergeJoin` with `Exchange hashpartitioning`.

**Q10 — RFM segmentation.** Per customer compute Recency (days since last purchase relative to a fixed reference date = max date in dataset), Frequency (distinct invoice count), Monetary (total spend). Bucket each into quintiles via `NTILE(5)`. Apply rule-based classification:

- Champions: R ≥ 4 AND F ≥ 4 AND M ≥ 4
- Loyal: R ≥ 3 AND F ≥ 3 AND M ≥ 3 AND not Champion
- At-Risk: R ≤ 2 AND F ≥ 3
- Lost: R ≤ 2 AND F ≤ 2

Output: count per segment plus the top 5 customers per segment ranked by Monetary score.

### Three-API implementation requirement

Each of the 10 queries is implemented in three places:
- `src/rdd/qNN_*.py` using only `sparkContext.textFile` + `map` / `reduceByKey` / `groupByKey` / `sortBy` etc. No DataFrames or SQL.
- `src/dataframe/qNN_*.py` using DataFrame API (`select`, `filter`, `groupBy`, `agg`, `Window`).
- `src/sql/qNN_*.py` using `spark.sql("…")` against registered temp views.

Total: **30 implementations**. All three implementations of each query produce semantically identical results; the benchmark harness verifies row counts agree.

## 5. Optimization & Benchmark Plan

### A. Plan capture

For all 20 DataFrame and SQL implementations, `capture_explain.py` calls `df.explain(mode="extended")` and writes the four sections (parsed logical, analyzed logical, optimized logical, physical) to `outputs/explain/qNN_<api>.txt`. The report shows the plans for at least Q1, Q5, Q8, Q9, Q10.

### B. Broadcast vs Sort-Merge Join (Q8 vs Q9)

- Q8 uses `broadcast()` hint → physical plan shows `BroadcastHashJoin`.
- Q9 is a self-join on a wide table → physical plan shows `SortMergeJoin` with `Exchange hashpartitioning`.
- The report places both physical plans side by side and reports execution times.

### C. CSV vs Parquet (Q3)

- Run Q3 on raw CSV; record time and bytes-read.
- Run Q3 on Parquet (partitioned by `Year`); record time and bytes-read.
- Report shows the delta and quotes the `FileScan` lines from each plan.

### D. Partition Pruning

- Variant of Q3 with `WHERE Year = 2010` against the year-partitioned Parquet.
- Verify physical plan shows `PartitionFilters: [isnotnull(Year#…), (Year#… = 2010)]` and only one partition directory is scanned.

### E. Caching

- Compute the shared `customer_totals` aggregate (used by Q6, Q7, Q10).
- Run Q6 cold → record time.
- Call `.cache()` + trigger materialization → run Q6 again → record time.
- Report the speedup.

### Benchmark harness — `src/benchmark/run_all.py`

For each of the 30 implementations capture:

| Metric | Source |
|---|---|
| Wall-clock time | `time.perf_counter()` around the action |
| Result row count | `count()` for DF/SQL, `len(collect())` for RDD |
| Stage and shuffle metrics | `spark.sparkContext.statusTracker()` + Spark event log JSON |
| Peak executor memory | Parsed from the event log |

Each query is run 3 times and the **median** is reported (a brief warm-up run is discarded). Output goes to `outputs/metrics/performance.csv` with columns: `query_id, api, runtime_seconds, result_rows, shuffle_read_mb, shuffle_write_mb, peak_memory_mb`.

`make_report_tables.py` reads that CSV and emits Markdown tables and PNG bar charts that are embedded in `REPORT.md`.

### Scalability tests (single-machine substitute for "different cluster sizes")

Vary two configuration knobs and re-run Q5 (window) and Q9 (sort-merge join):

- `spark.sql.shuffle.partitions` ∈ {4, 8, 50, 200}
- `spark.master` ∈ {`local[1]`, `local[4]`, `local[*]`}

Report a single scalability table with runtime across all combinations.

### Honest caveats called out in the report

- Wall-clock times on one laptop are noisy → we run 3× and take the median.
- Spark event-log memory metrics are approximate → the report says so explicitly.
- RDD implementations of windowed and ranked queries (Q5, Q6, Q10) are deliberately verbose → that *is* the point: they motivate Catalyst.

## 6. The Report — `REPORT.md`

Mirrors the deliverable list in the assignment:

1. Problem Statement (from §1 above)
2. Dataset Description (from §2)
3. Queries Implemented — for each query: SQL code, DataFrame code, RDD code, plus a screenshot or text excerpt of `.explain()`
4. Execution Plan Comparison — logical/optimized/physical plans for representative queries; commentary on what Catalyst did
5. Performance Comparison Table — auto-generated from `performance.csv`; identifies the fastest API per query and explains why
6. Final Insights — analytical conclusions from Q10 (RFM) and Q3/Q5 (trends), plus lessons learned about Spark optimization

The report is written as we go (not at the end), so each query's section is filled in once that query's three implementations and benchmark are done.

## 7. Out of Scope

- Predictive modeling (clustering, regression). The rubric example mentions "prediction" as one option, but the chosen research question is descriptive segmentation; we deliberately do *not* train ML models.
- Streaming. Dataset is bounded.
- Real cluster deployment. We approximate it via varying `local[N]` and `shuffle.partitions`.
- A `.docx` version of the report. Markdown only; the user converts at the end if needed.

## 8. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| RDD implementation of window functions (Q5, Q6, Q10) is hard to write correctly | Use `groupBy` + `sortBy` + index-based slicing; verify row counts match the DataFrame implementation |
| Spark event log parsing for memory metrics is fragile | Fall back to `statusTracker()` stage info if event-log parse fails; mark column as "approx" in the report |
| The 95 MB CSV is too small to show dramatic distributed-execution benefits | Keep the dataset; explain in the report that the *workload pattern* (high-cardinality group-bys, windows, joins) is the point, not raw size |
| Benchmark variance on a laptop hides real speedups | Median-of-3 runs; warm-up run discarded |

## 9. Success Criteria

- All 30 query implementations run end-to-end without error.
- All three implementations of each query agree on row count.
- `outputs/explain/` has 20 files; `outputs/metrics/performance.csv` has 30 rows.
- `REPORT.md` covers all 6 deliverable sections, includes plan excerpts and the performance table.
- Optimization demos (B, C, D, E) each show a measurable speedup or a plan difference visible in `.explain()`.
